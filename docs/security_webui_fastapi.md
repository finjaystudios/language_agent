# Security Plan: Chainlit Web UI to FastAPI

## Scope

This plan covers service-to-service authentication between the Chainlit Web UI
and the FastAPI backend. The applications are separate runtime services and are
expected to run as separate Docker containers. Browser users interact with
Chainlit only; Chainlit server-side code calls FastAPI over HTTP.

This feature should not add browser user login, OAuth, reverse proxy
configuration, or TLS termination.

## Current Connection Audit

FastAPI currently exposes:

| Method | Path | Current purpose | Planned access |
| --- | --- | --- | --- |
| `GET` | `/health` | Lightweight process health check. Does not initialise the LLM. | Public |
| `GET` | `/` | Service metadata for local tooling and discovery. | Public |
| `POST` | `/api/chat` | Full response chat endpoint. May route intent and trigger LLM work. | Protected |
| `POST` | `/api/chat/stream` | Server-Sent Events streaming endpoint. Triggers LLM work for streaming modes. | Protected |

The Chainlit Web UI currently calls FastAPI from `webui/client.py`:

| Client method | FastAPI request | Planned auth behaviour |
| --- | --- | --- |
| `FastAPIClient.health()` | `GET /health` | No auth header required |
| `FastAPIClient.chat_full()` | `POST /api/chat` | Send service API key |
| `FastAPIClient.chat_stream()` | `POST /api/chat/stream` | Send service API key |

The Chainlit browser UI should never receive the FastAPI API key. The key stays
in the Chainlit server process environment and is attached only by the
server-side HTTP client.

## Chosen Authentication Mechanism

Use a static shared service API key:

- Header: `X-API-Key`
- FastAPI secret env var: `FASTAPI_API_KEY`
- Chainlit secret env var: `FASTAPI_API_KEY`
- Auth toggle env var: `AUTH_ENABLED`

`X-API-Key` is preferred here because this is not user authentication and does
not need JWT claims, token refresh, scopes, or identity provider integration.
The header is explicit, easy to test with Bruno and curl, and avoids implying a
user bearer token model.

FastAPI should compare the inbound header value to `FASTAPI_API_KEY` with
constant-time comparison, for example `secrets.compare_digest()`.

## Protected Endpoints

Protect every endpoint that can perform LLM-backed work or reveal model-backed
responses:

- `POST /api/chat`
- `POST /api/chat/stream`

These endpoints cover full response generation, intent-routed requests, chat
requests, and streaming requests.

## Public Endpoints

Keep these unauthenticated:

- `GET /health`
- `GET /`

`/health` must remain public so Docker `HEALTHCHECK`, local smoke checks, and
orchestration probes continue to work without a secret. `/` returns only basic
service metadata and can remain public unless future metadata becomes sensitive.

OpenAPI routes are generated by FastAPI. This plan does not require hiding
`/docs` or `/openapi.json`; if deployments need stricter API discovery controls,
that should be handled as a separate hardening task.

## Environment Variables

FastAPI:

| Variable | Example | Required when auth is enabled | Purpose |
| --- | --- | --- | --- |
| `AUTH_ENABLED` | `true` | Yes | Enables API key validation on protected routes. |
| `FASTAPI_API_KEY` | `change-me-local-dev-only` | Yes | Shared service secret accepted by FastAPI. |
| `CORS_ALLOWED_ORIGINS` | `http://localhost:8001,http://127.0.0.1:8001` | No | Optional explicit browser origins allowed to call FastAPI directly. |

Web UI:

| Variable | Example | Required when auth is enabled | Purpose |
| --- | --- | --- | --- |
| `FASTAPI_BASE_URL` | `http://localhost:8000` | Yes | Backend base URL used by Chainlit server-side client. |
| `FASTAPI_API_KEY` | `change-me-local-dev-only` | Yes | Shared service secret sent by Chainlit to FastAPI. |

Do not commit real API keys. `.env.example` may contain placeholder values only.
Developer-specific values belong in `.env`, shell environment, Docker secrets,
or deployment-specific secret management.

## Local Development Behaviour

Recommended local behaviour:

- Default `AUTH_ENABLED=true` in examples so developers exercise the secured
  path by default.
- Allow `AUTH_ENABLED=false` only for focused local debugging and tests that
  explicitly need unauthenticated protected routes.
- If `AUTH_ENABLED=true` and `FASTAPI_API_KEY` is missing or blank, FastAPI
  should fail closed for protected endpoints with a sanitized `500` or startup
  configuration error. It should not silently disable auth.
- Chainlit should attach `X-API-Key` only when `FASTAPI_API_KEY` is set. Missing
  or incorrect keys should produce a normal backend HTTP error in the Web UI.
- The CLI should remain unaffected because it uses local services directly and
  does not call FastAPI over HTTP.

## Docker Behaviour

FastAPI backend container:

- Receives `AUTH_ENABLED=true`.
- Receives `FASTAPI_API_KEY` from the runtime environment, not from the image.
- Keeps the existing Docker `HEALTHCHECK` against `GET /health` unchanged.

Web UI container:

- Receives `FASTAPI_BASE_URL` pointing at the FastAPI service URL.
- Receives the same `FASTAPI_API_KEY` value as the backend.
- Sends `X-API-Key` from server-side Chainlit code only.

Do not bake secrets into Dockerfiles or images. Do not expose the API key via
client-side JavaScript, Chainlit message metadata, logs, or browser-visible
configuration.

## Testing Strategy

Backend unit/API tests:

- Assert `GET /health` succeeds without `X-API-Key`.
- Assert `GET /` succeeds without `X-API-Key`.
- Assert `POST /api/chat` returns `401` or `403` without a valid key when
  `AUTH_ENABLED=true`.
- Assert `POST /api/chat/stream` returns `401` or `403` without a valid key
  when `AUTH_ENABLED=true`.
- Assert both protected endpoints accept the configured `X-API-Key`.
- Assert `AUTH_ENABLED=false` preserves existing unauthenticated behaviour for
  local compatibility tests.

Web UI client tests:

- Assert `FastAPIClient.health()` does not require or send the API key.
- Assert `FastAPIClient.chat_full()` sends `X-API-Key` when configured.
- Assert `FastAPIClient.chat_stream()` sends `X-API-Key` when configured.
- Assert backend `401` or `403` responses are surfaced as readable UI errors.

Browser automation tests:

- Update the fake backend to accept or assert the `X-API-Key` header on chat
  endpoints.
- Ensure no browser-visible text, metadata, or network request from the browser
  contains the FastAPI API key.
- Assert a wrong Web UI key produces a readable authentication failure.
- Assert a missing Web UI key produces a readable configuration failure before
  calling FastAPI.

CORS tests:

- Assert no CORS headers are emitted when `CORS_ALLOWED_ORIGINS` is unset.
- Assert configured origins receive CORS headers.
- Assert unconfigured origins are rejected.

Bruno:

- Add an environment variable such as `apiKey`.
- Add `X-API-Key: {{apiKey}}` to protected chat requests only.
- Keep system health checks unauthenticated.

## Known Limitations

- A static API key authenticates the Web UI service, not individual users.
- It does not provide authorization scopes, per-user audit trails, token expiry,
  or revocation without rotating the shared secret.
- HTTP traffic is not encrypted by this plan. Production deployments should run
  behind infrastructure that provides HTTPS/TLS.
- Anyone with access to the Chainlit runtime environment can read the shared
  service key.
- This plan does not protect FastAPI from direct network access by itself; it
  should be paired with normal deployment network controls.
- CORS is not part of the current Chainlit chat path because the browser does
  not call FastAPI directly. Optional CORS support is limited to explicit
  origins for future browser-origin flows.

## Future Upgrade Path

If the product later needs user accounts, per-user quotas, auditability, or
third-party identity, upgrade from the shared service key to JWT or OAuth:

- Keep service-to-service auth at the Chainlit-to-FastAPI boundary if Chainlit
  remains the trusted backend-for-frontend.
- Add user identity in Chainlit first, then pass validated user context to
  FastAPI as signed JWT claims or a separate internal authorization context.
- Introduce scopes or roles only when there are multiple protected capabilities.
- Maintain `/health` as unauthenticated unless the deployment platform supports
  authenticated health probes.
